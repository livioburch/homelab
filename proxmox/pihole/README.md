# 🕳️ Pi-hole – Network-Wide DNS & Telemetry Hardening

This directory documents the deployment and configuration of Pi-hole, which serves as the primary upstream DNS server and ad-blocker for my entire homelab infrastructure.

## ⚙️ Infrastructure Integration

Unlike the application microservices, Pi-hole is deployed as a dedicated, lightweight **Linux Container (LXC)** directly on top of Proxmox VE (`pve01`). 

* **OS / Environment:** Debian 11 minimal footprint instance.
* **Resource Allocation:** 512 MB RAM / 1 vCPU / 8 GB Storage (optimized for high-throughput DNS caching).

## 🛡️ Core Functionalities inside the Lab

* **Telemetry & Tracking Mitigation:** Automatically drops outgoing diagnostic and tracking metrics generated by various smart devices, operating systems, and testing environments.
* **Local DNS Resolution (Split-Brain DNS):** Manages local domain name pointers, allowing production VMs and Docker containers to resolve internal infrastructure names (e.g., `proxmox.local`, `vaultwarden.local`) without relying on external routing loops.
* **Upstream Protection:** Configured to forward verified queries to privacy-respecting upstream providers (e.g., Cloudflare/Quad9) utilizing secure DNS protocols.

## 🔄 High-Availability & Fallback Design

To prevent a "chicken-and-egg" bootstrapping failure (where Docker containers fail to boot after a power outage because the DNS container hasn't fully initialized), a dual-resolver strategy is implemented across the lab:

1. **Primary DNS:** All local container stacks prioritize this Pi-hole instance for full security filtering and local hostname lookups.
2. **Secondary Fallback (`1.0.0.1`):** A high-availability external public resolver is defined as a backup. If the Pi-hole LXC is undergoing maintenance or restarting, core infrastructure stacks can still resolve external dependencies without downtime.
